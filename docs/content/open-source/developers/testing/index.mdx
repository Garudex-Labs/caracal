The test suite is broad in directory coverage, but implementation depth is
uneven. Treat the tree as the current testing map, not proof that every
subsystem has complete behavioral coverage.

**Audience:** Developer

**Context:** Use this page when deciding where a test belongs, which markers to use, and what existing coverage areas already exist.

## Core Explanation

The repository test tree is intentionally split by test depth:

| Area | Purpose |
| --- | --- |
| `tests/unit` | isolated module and command tests |
| `tests/integration` | multi-component behavior |
| `tests/e2e` | end-to-end workflows |
| `tests/security` | abuse, fuzzing, and regression coverage |
| `tests/sdk` | SDK-specific placeholders and future coverage |
| `tests/validate_*.py` | test-infrastructure validation scripts |

Markers defined in `pyproject.toml`:

- `unit`
- `integration`
- `e2e`
- `security`
- `property`
- `asyncio`
- `slow`

## Practical Usage

Common runs:

```bash
pytest -m unit
pytest -m integration
pytest -m security
pytest --cov=caracal --cov-report=term-missing
```

The GitHub Actions test workflow currently runs one Python 3.12 pytest job:

```bash
uv run pytest -n auto --cov=caracal --cov-report=term-missing --cov-report=xml --cov-report=html --junitxml=junit.xml
```

CI collects the whole pytest suite under `tests/` and publishes terminal, XML,
HTML, Codecov, and JUnit outputs. It does not currently split unit,
integration, security, and e2e tests into separate CI jobs, and it does not
enforce a coverage percentage threshold.

Infrastructure validation:

```bash
python tests/validate_all.py
python tests/simulate_ci.py
```

## Internal Behavior

The suite is organized around repository subsystems, not just test types. There are dedicated areas for:

- CLI
- core authority logic
- DB and migrations
- deployment
- Flow
- MCP
- Merkle
- monitoring
- provider and Redis behavior

This structure reflects the repository layout and should remain aligned with it.
Coverage is strongest where concrete test files exist under unit/core,
unit/cli, unit/db, unit/deployment, unit/runtime, unit/mcp, integration/core,
integration/mcp, and a small set of e2e workflow tests. Some directories exist
mainly to reserve the intended shape of the suite.

## Writing Strategy

Place tests where they match both:

- the behavioral depth
- the subsystem under test

Examples:

- pure validation logic belongs in `tests/unit/core`
- database-backed authority flows belong in `tests/integration/core`
- full command workflows belong in `tests/e2e/cli`
- attack or malformed-input cases belong in `tests/security`

## Edge Cases And Constraints

- Some e2e and broader coverage areas are still placeholders. Do not assume a directory implies complete behavioral coverage.
- Some security fuzzing, abuse, and integration files still contain placeholder `pass` tests or smoke-level assertions.
- SDK coverage is currently Python-focused; TypeScript SDK space exists as a placeholder.
- Monitoring, storage, and several UI-oriented directories are structurally present but thin or empty.
- Coverage reporting in the repo is configured broadly, but CI does not currently enforce the long-term 90% target.
