# Authority Enforcement Model

Authority enforcement in Caracal is pre-execution decision gating: actions are evaluated against explicit authority context before side effects run.

## Plain-language explanation

Caracal treats delegated action execution as a security boundary. The model is built so an operation is not considered valid just because it is syntactically correct or technically reachable. It must also be authorized in the right scope, with the right identity context, and with controls that fail closed when required context is missing.

In practical terms, the model connects these elements:

- identity and principal context
- policy constraints
- mandate/delegation scope
- runtime verification checks
- audit reconstruction capability

## Where this appears in the system

- Host entrypoint orchestration and runtime command boundaries (`caracal` host mode vs in-container mode).
- Runtime preflight and hard-cut checks that reject unsafe configuration in protected paths.
- CLI and SDK operations where authority-relevant identifiers are created, mutated, and validated.
- Threat analysis and incident triage where decisions must be replayable from logs and state.

## Control properties

1. Pre-execution enforcement
   A request path should reject invalid authority context before provider-side action execution.
2. Fail-closed behavior
   Missing critical security context does not degrade to permissive execution in protected runtime paths.
3. Canonical scope handling
   Workspace and identity scope must remain explicit through command/API boundaries.
4. Auditability
   Decisions and transitions should be reconstructable for incident review.

## Constraints and edge cases

- Scope confusion is a primary failure mode: if principal/workspace/policy context is ambiguous, deny and require explicit context.
- Configuration drift can invalidate security posture: runtime checks should reject prohibited modes rather than warn.
- Cross-workspace assumptions are invalid unless explicitly modeled by authority primitives.
- Provider availability is not equivalent to authorization success; connectivity checks do not replace policy enforcement.
