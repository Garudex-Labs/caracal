# Quickstart

## Outcome

Bring up a local Caracal runtime, open Flow, verify health through host commands, and execute one SDK connectivity check in under ten minutes.

## Prerequisites

- Python 3.10+ with `caracal-core` available in the active environment.
- Docker with Compose support (`docker compose`).
- Network access to pull runtime images when running `caracal up` without `--no-pull`.

## Step 1: Verify host command and Docker Compose

Run:

```bash
caracal --help
docker compose version
```

Verification criteria:

- `caracal --help` prints subcommands.
- `docker compose version` returns successfully.

If either command fails, fix host environment setup before proceeding.

## Step 2: Create the secure local runtime env

Run:

```bash
export CCL_HOME="${CCL_HOME:-$HOME/.caracal}"
mkdir -p "$CCL_HOME/runtime"
umask 077
cat > "$CCL_HOME/runtime/.env" <<EOF
CCL_DB_PASSWORD=$(openssl rand -hex 24)
CCL_REDIS_PASSWORD=$(openssl rand -hex 24)
CCL_VAULT_TOKEN=dev-local-token
VAULT_AUTH_SECRET=$(openssl rand -hex 32)
VAULT_ENC_KEY=$(openssl rand -hex 16)
CCL_ENV_MODE=dev
CCL_ALLOW_INTERNAL_PROVIDER_URLS=true
EOF
chmod 600 "$CCL_HOME/runtime/.env"
```

Expected behavior:

- `$CCL_HOME/runtime/.env` exists with owner-only permissions.
- Redis, database, local vault bootstrap secrets, and explicit dev relaxations are set before Compose starts.

## Step 3: Bootstrap and start runtime services

```bash
caracal bootstrap
caracal up
```

Expected behavior:

- Bootstrap creates the system principal, grants admin/dev setup capabilities, and issues the first AIS nonce.
- Runtime stack starts with `postgres`, `redis`, `vault`, and `mcp` services.
- Compose startup completes without non-zero exit codes.

Optional variant:

```bash
caracal up --no-pull
```

Use `--no-pull` only when required images are already present locally.

## Step 4: Mint a session token

Run:

```bash
eval "$(caracal auth token --format env)"
```

Verification criteria:

- `CCL_SESS_TOKEN` is set in the current shell.
- The token was minted through the local AIS Unix socket.

## Step 5: Open Flow (TUI)

Run:

```bash
caracal flow
```

Expected behavior:

- Flow launches inside the runtime container.
- You can navigate screens without immediate runtime connection errors.

`caracal` with no args also routes to Flow by default on host entrypoints.

## Step 6: Validate runtime logs and command boundary

Run:

```bash
caracal logs -f
```

Optional service filtering:

```bash
caracal logs postgres redis mcp
```

Verification criteria:

- Log stream is active.
- No repeated fatal startup failures.

Then open restricted container CLI:

```bash
caracal cli
```

You are now in the in-container command boundary for authority operations.

## Step 7: Validate SDK connectivity baseline (Python)

Minimal SDK check:

```python
import os
from caracal_sdk.client import CaracalClient

client = CaracalClient(api_key=os.environ["CCL_SESS_TOKEN"])
print("client-initialized")
client.close()
```

Verification criteria:

- Client initialization succeeds when the AIS-issued session token and runtime URL are valid.
- No `SDKConfigurationError` is raised.

The SDK resolves base URL from `CCL_API_URL` or defaults to `http://localhost:${CCL_API_PORT:-8080}`.

## Common failure branches

- `caracal up` fails: Docker/Compose unavailable, permissions missing, or image pull/network issues.
- `caracal flow` fails immediately: runtime services not healthy yet.
- SDK init fails: missing/invalid session token or incorrect runtime URL/port mapping.
