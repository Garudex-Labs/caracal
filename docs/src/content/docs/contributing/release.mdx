Caracal uses only three versioning modes: dev, rc, and stable. rc and stable artifacts use the same artifact destinations; dev artifacts stay local.

---

## Version scheme

| Mode | Purpose | OCI/binary example | npm example | PyPI example |
| --- | --- | --- | --- | --- |
| dev | Development builds | `2026.05.14-dev.shaabc1234` | `1.4.2-dev.shaabc1234` | not published |
| rc | rc builds | `2026.05.14-rc.shaabc1234` | `1.4.2-rc.shaabc1234` | `1.4.2rc0+shaabc1234` |
| stable | Released production versions | `2026.05.14` | `1.4.2` | `1.4.2` |

The PyPI rc form is PEP 440 compatible while preserving the same rc/sha meaning used by npm, OCI tags, binaries, and GitHub tags.

Git tags use a leading `v`:

- stable: `vYYYY.MM.DD[.N]`
- rc: `vYYYY.MM.DD[.N]-rc.<number>` or `vYYYY.MM.DD[.N]-rc.sha<gitsha>`

The tag drives binaries, OCI image tags, GitHub Releases, Helm `appVersion`/image values, and release manifests. Package metadata drives npm and PyPI publishing.

## rc workflow

Use rc when a downstream project must consume Caracal as a normal third-party dependency before stable. The consumer changes only package versions and registry/index endpoints; it must not reference Caracal source paths, workspace links, editable installs, or Docker build contexts from the Caracal checkout.

```sh
scripts/rc.sh prepare
git add -A
git commit -m "rc: vYYYY.MM.DD-rc.sha<gitsha>"
git tag -a vYYYY.MM.DD-rc.sha<gitsha> -m vYYYY.MM.DD-rc.sha<gitsha>
git push origin HEAD
git push origin vYYYY.MM.DD-rc.sha<gitsha>
```

`scripts/rc.sh prepare` writes `releases/<rc-tag>/manifest.json`, stamps Helm metadata, stamps npm package versions as `<base>-rc.sha<gitsha>`, and stamps PyPI versions as `<base>rc0+sha<gitsha>`. Use `scripts/rc.sh version` only when you need to preview the manifest without stamping package or Helm metadata. Pushing the rc tag runs the release workflow, publishes GHCR images, and creates a GitHub Release for rc.

Publish packages after `prepare`:

```sh
./scripts/publishNpm.sh       # changed rc packages publish with dist-tag rc
./scripts/publishPypi.sh      # changed rc packages use PEP 440 rc metadata
```

Switch a downstream project between rc and stable by changing only the Caracal versions it already consumes:

```sh
npm install @caracalai/sdk@1.4.2-rc.sha<gitsha>   # rc
npm install @caracalai/sdk@1.4.2                  # stable
uv add caracalai-sdk==1.4.2rc0+sha<gitsha>        # rc
uv add caracalai-sdk==1.4.2                       # stable
```

For containers, change only `ghcr.io/garudex-labs/caracal-{svc}:v<version>`. For binaries, install the desired `--version`. Consumers must not add Caracal source paths, editable installs, Docker build contexts from this checkout, or extra env files.

## stable workflow

Stable releases are created from `main` with the release script:

```sh
scripts/release.sh
scripts/release.sh --dry-run
```

The script applies pending changesets, stamps Helm metadata, writes `releases/<tag>/manifest.json`, tags the release, and pushes `main` plus the tag. The tag triggers `release.yml` in GitHub Actions. The stable pipeline:

1. Verifies the pushing actor is listed in `.github/MAINTAINERS`.
2. Builds runtime/console binaries and container images in parallel.
3. Pushes immutable image tags to GHCR with SLSA provenance and SBOM.
4. Waits for the protected `release-approval` environment.
5. Creates a GitHub Release with generated release notes, binaries, checksums, and provenance attestations.
6. Runs release validation for rc and stable tags.
7. Promotes stable images to the `vYYYY.MM` series tag and `latest` only after validation passes.

Only maintainers cut stable releases. Do not push CalVer tags from feature branches.

### Release approval and repository protection

Stable releases are production changes. The `release-approval` environment must require at least one approving maintainer who did not push the release tag. The approval confirms that the tag points to a commit reachable from `main`, the manifest is present under `releases/<tag>/manifest.json`, pending changesets have been applied, and release notes describe migrations, compatibility changes, and rollback constraints.

Configure repository protection with:

| Control | Required setting |
| --- | --- |
| Stable release environment | `release-approval` with required reviewers |
| RC release environment | `rc-release`; reviewer protection is optional |
| PyPI publish environment | `pypi` with required reviewers |
| npm publish environment | `npm-publish` with required reviewers |
| Protected tags | `v*`; block deletion and force-push |
| Release branches | `main` requires pull request review and required status checks |

The actor who pushes a stable tag must not approve the `release-approval`, `pypi`, or `npm-publish` environment for that release. If an emergency requires bypassing a protection rule, record the reason in the release notes and run the full validation workflow before announcing the release as stable.

## Shell, Console, and Console binaries

The shell (`caracal`) and Console (`caracal-console`) are compiled to self-contained binaries using `bun --compile`. Ten archives are produced per release (five targets x two binaries), each containing one executable:

| Target | Shell archive | Console archive |
| --- | --- | --- |
| `linux-amd64` | `caracal-shell-linux-amd64-vYYYY.MM.DD.tar.gz` | `caracal-console-linux-amd64-vYYYY.MM.DD.tar.gz` |
| `linux-arm64` | `caracal-shell-linux-arm64-vYYYY.MM.DD.tar.gz` | `caracal-console-linux-arm64-vYYYY.MM.DD.tar.gz` |
| `darwin-amd64` | `caracal-shell-darwin-amd64-vYYYY.MM.DD.tar.gz` | `caracal-console-darwin-amd64-vYYYY.MM.DD.tar.gz` |
| `darwin-arm64` | `caracal-shell-darwin-arm64-vYYYY.MM.DD.tar.gz` | `caracal-console-darwin-arm64-vYYYY.MM.DD.tar.gz` |
| `windows-amd64` | `caracal-shell-windows-amd64-vYYYY.MM.DD.zip` | `caracal-console-windows-amd64-vYYYY.MM.DD.zip` |

Archives are attached to the GitHub Release with `manifest.json`, release installer copies, a `SHA256SUMS` file covering all release assets, and SLSA provenance attestations. rc tags create GitHub Releases for rc; stable tags create GitHub Releases for stable.

### Install

The Runtime and Console have separate install targets:

```sh
# Linux/macOS: Console
curl -fsSL https://raw.githubusercontent.com/Garudex-Labs/caracal/main/install-console.sh | sh

# Windows: Console
iwr -useb https://raw.githubusercontent.com/Garudex-Labs/caracal/main/install-console.ps1 | iex
```

The installers detect OS and architecture, resolve the latest release tag from the GitHub API (or accept `--version`/`-Version`), download the matching shell and selected tool archives, verify them against `SHA256SUMS`, optionally verify GitHub artifact provenance, extract them, and place the binaries on `PATH`.

### Upgrade

Re-run the same install command. The installer overwrites the binary in place with the new version.

### Uninstall

Linux/macOS: remove `caracal` and `caracal-console` from `~/.local/bin`.

Windows: delete `%LOCALAPPDATA%\Programs\caracal` and remove the user PATH entry.

Runtime and service container images are published to GitHub Container Registry on every release:

| Image | Service |
| --- | --- |
| `ghcr.io/garudex-labs/caracal-runtime` | Runtime shell image |
| `ghcr.io/garudex-labs/caracal-api` | Control-Plane API |

Stable releases promote the same image manifests to the `vYYYY.MM` series and `latest` tags only after `release-approval` and release validation pass. rc releases never move `latest`.

## Helm chart metadata

The Helm chart uses SemVer-compatible `Chart.yaml` versions and CalVer application versions. Release automation stamps:

| Field | Stable example | Meaning |
| --- | --- | --- |
| `Chart.yaml version` | `2026.5.14` | Helm package version; numeric components are normalized for SemVer compatibility |
| `Chart.yaml appVersion` | `2026.05.14` | Caracal release version without the leading `v` |
| `values.yaml global.tag` | `2026.05.14` | Image tag suffix; templates render images as `v<global.tag>` |
| `manifest.json helm.imageTag` | `2026.05.14` | Source of truth for Helm image alignment |

Do not edit chart versions by hand during a release. `scripts/release.sh` and `scripts/rc.sh prepare` stamp the chart and manifest from the same release identity. The release workflow rejects tags when `Chart.yaml`, `values.yaml`, and `manifest.json` disagree.

Validate committed manifests locally:

```sh
node scripts/validateReleaseManifest.mjs
```

## npm and PyPI packages

npm and PyPI package publishing must run through protected release automation for stable releases. The local scripts in `scripts/` are maintainer tools for package selection, local dry-runs, and emergency recovery; they must not be the normal stable publishing path. Each protected publish workflow diffs the current `HEAD` against the latest reachable release tag and selects only changed publishable package directories by default. Use `--base <ref>` to override the diff base, `--select` to manually narrow the detected set, or `--all` only for an intentional full-package publish. npm rc versions publish with the `rc` dist-tag; stable versions publish with `latest`.

```sh
gh workflow run publishNpm.yml -f package=changed -f dryRun=true
gh workflow run publishNpm.yml -f package=changed
gh workflow run publishPypi.yml -f package=changed -f dryRun=true
gh workflow run publishPypi.yml -f package=changed
```

The local scripts refuse stable publishing unless `CARACAL_ALLOW_LOCAL_STABLE_PUBLISH=1` is set. Use that override only for approved emergency recovery after recording the reason in the release notes.

**npm packages** (`@caracalai/*`): `core`, `oauth`, `admin`, `identity`, `revocation`, `sdk`, `transport-mcp`, `transport-a2a`, `mcp-express`, `mcp-fastmcp`, `tokenstate-postgres`, `revocation-redis`. Browse at [npmjs.com/~caracal-run](https://www.npmjs.com/~caracal-run).

**PyPI packages** (`caracalai-*`): `core`, `oauth`, `identity`, `revocation`, `sdk`, `transport-mcp`, `mcp-fastmcp`, `revocation-redis`. Browse at [pypi.org/user/CaracalAI](https://pypi.org/user/CaracalAI).

## SLSA provenance

The release pipeline generates SLSA Level 3 provenance attestations for binaries and container images. Verify an artifact:

```sh
gh attestation verify caracal-console-linux-amd64-v2026.05.12.tar.gz --repo garudex-labs/caracal
```

High-assurance installs fail closed when provenance cannot be verified:

```sh
curl -fsSLO https://github.com/Garudex-Labs/caracal/releases/download/v2026.05.12/install-console.sh
sh install-console.sh --version v2026.05.12 --require-provenance
```

```powershell
.\install-console.ps1 -Version v2026.05.12 -RequireProvenance
```

## Hotfixes

For a critical fix on an already-released version:

1. Branch from the release tag: `git checkout -b hotfix-jti v2026.05.12`
2. Apply the fix and open a PR against `main`.
3. Merge after review; cherry-pick to `main` if needed.
4. Tag the hotfix: `git tag v2026.05.12.1 && git push origin v2026.05.12.1`.

Tags must point to commits reachable from `main`.

## Post-release validation

Every release tag has a directory under `caracal/releases/<tag>/` that owns all of its metadata:

- `manifest.json`: pins every artifact (binaries, containers, Helm metadata, PyPI, npm) to a mode and version, committed alongside the changeset version bump or rc preparation commit.
- `../schema/manifest.schema.json`: documents the manifest shape required for new releases.
- `validation.md`: the aggregated post-release validation report.
- `findings/*.jsonl`: raw per-area findings.

The `.github/workflows/postReleaseValidation.yml` workflow runs inside `release.yml` for rc and stable tags, or independently via `workflow_dispatch`, and opens a PR that adds `validation.md` and `findings/` to that directory. Blocker findings fail release validation and prevent stable image promotion.

Run a single check locally:

```sh
CARACAL_RELEASE=v2026.05.12 FINDINGS_DIR=/tmp/findings \
  bash caracal/scripts/postRelease/validateRegistryMetadata.sh
```
