Caracal publishes a layered set of packages across TypeScript, Python, and Go. The main SDK packages are the default way to run agents through Caracal and call the Gateway with scoped access. Companion packages handle token exchange, identity verification, revocation, and administration. Connector packages adapt Caracal to direct-verifier and adapter paths when the Gateway is not the right boundary.

Start with an SDK plus the Gateway. Install identity, revocation, transport, or connector packages directly only when you are protecting a service you own or building a certified adapter for a provider protocol the Gateway cannot proxy.

## Shared concepts

**Context propagation** — The SDK packages bind agent run and delegated-permission identity to the execution context (AsyncLocalStorage in TypeScript, context vars in Python, `context.Context` in Go). Gateway-bound outbound calls within that context automatically carry the correct `Authorization`, `traceparent`, and `baggage` headers.

**Package layering** — The main SDKs own agent lifecycle and context propagation. TypeScript and Go use their OAuth packages for client-secret token exchange; Python implements its SDK token-source flow in `caracalai-sdk` and depends on `caracalai-identity`. Install companion packages directly when building a server that verifies mandates, a custom STS exchange client, or a control-plane tool.

**Environment variables** — All SDK clients read `CARACAL_COORDINATOR_URL`, `CARACAL_ZONE_ID`, `CARACAL_APPLICATION_ID`, and `CARACAL_SUBJECT_TOKEN` via their `fromEnv()` / `from_env()` / `FromEnv()` constructors.
