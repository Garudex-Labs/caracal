Caracal ships as one versioned set of six container images with a single release train. Every deployment profile runs the same images and the same policy, audit, and credential behavior; profiles differ only in where the containers run, who operates the data stores, and how much surrounding platform you attach. Kubernetes is not required: a single Docker host is a supported production posture, and the orchestrated tiers exist for availability and scale rather than for capability.

## Profile Map

| Profile             | Scale                       | Path                                                                                                                                         | Data stores                                  |
| ------------------- | --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| Evaluation          | Laptop or shared dev box    | [Install the CLI](/get-started/install-caracal/) and run `caracal up`                                                                        | Bundled Postgres and Redis containers        |
| Single VM           | One team, production        | Installed CLI on any Docker host, optionally provisioned with the [`caracalHost` OpenTofu module](/operations/opentofu/#single-vm-bootstrap) | Bundled containers plus scheduled backups    |
| Lightweight cluster | Small production Kubernetes | [Helm chart](/operations/kubernetes-helm/) in `rc` or `stable` mode                                                                          | Entry-tier managed Postgres and Redis        |
| Enterprise HA       | Multi-team production       | Helm `stable` mode via the [`caracalStack` OpenTofu root](/operations/opentofu/) with an external secret manager                             | Managed HA Postgres and Redis with PITR      |
| Hyperscale          | High-volume, multi-cluster  | Same chart with raised HPA ceilings, replay persistence, and topology spread; one release per cluster                                        | Data-platform-owned stores sized for streams |

## Evaluation

Run `caracal up` on any machine with Docker. The stack self-generates its secrets, binds every port to loopback, and reports readiness when all services pass their probes. Nothing is exposed beyond the host and `caracal purge` removes everything. Use this tier to evaluate policy behavior and SDK integration, not to hold real credentials long-term.

## Single VM

The installed CLI carries the production compose topology, pinned images, health probes, and `caracal upgrade` for no-maintenance-window upgrades. Runtime secrets are generated on the host at first start and never leave it. All service ports stay loopback-bound: publishing the web console or Gateway is a deliberate act through the reverse proxy or tunnel you already operate, which keeps TLS and network exposure under your control.

This tier is production-grade for a single team when you schedule [backups](/operations/backup-retention/) and treat the VM as replaceable: a fresh host plus the latest backup bundle restores the stack. To provision hosts declaratively, the [`caracalHost` module](/operations/opentofu/#single-vm-bootstrap) renders cloud-init user data that works with any VM resource on any cloud.

Move up when you need zero-downtime host maintenance, more than one replica per service, or organizational separation of data-store operations.

## Lightweight Cluster

The Helm chart deploys the same topology onto any conformant Kubernetes 1.30+. In `rc` mode the chart may still run in-cluster data stores for staging; in `stable` mode the chart's guardrails reject in-cluster Postgres and Redis, so pair it with entry-tier managed database and cache services. This is the smallest footprint that gives rolling upgrades with zero unavailable replicas and per-service scaling.

## Enterprise HA

Drive the chart through the [`caracalStack` OpenTofu root](/operations/opentofu/), pin released chart versions from the OCI registry, and deliver the runtime Secret with your secret manager - the External Secrets flow in [Deploy on Managed Kubernetes](/operations/cloud-reference-deployments/) is the reference shape. The [cloud profile contract](/operations/cloud-native-profiles/) defines what the managed Postgres, Redis, secrets, and observability services must provide. Applies are atomic, migration hooks roll back with the release, and NetworkPolicy defaults to closed egress.

## Hyperscale

The enterprise HA shape scales without new machinery: raise HPA replica ceilings, enable replay persistence, add topology spread across zones, and size Redis Streams and Postgres connection capacity with your data platform team. Run one release per cluster and shard by zone or region at the load balancer rather than stretching a single stack; see [Scale Capacity](/operations/scale-capacity/) for per-service scaling behavior.

## Ownership Boundaries

Every external dependency attaches through an open standard, so managed services from any provider plug in without Caracal-specific adapters.

| Component                   | Ownership                                         | Boundary                                                                                                                            |
| --------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| Caracal services            | Caracal                                           | Versioned images on one release train; `caracal upgrade` or Helm upgrades.                                                          |
| PostgreSQL                  | Bundled on VM tiers; yours on `stable` Kubernetes | Standard connection URLs meeting the [managed dependency contract](/operations/cloud-native-profiles/#managed-dependency-contract). |
| Redis                       | Bundled on VM tiers; yours on `stable` Kubernetes | Redis protocol with Streams; authentication and persistence per the same contract.                                                  |
| Secrets                     | Yours                                             | Self-generated files on hosts; a Kubernetes Secret delivered by any secret manager on clusters.                                     |
| Console sign-in             | Caracal, with your identity configuration         | OIDC providers and SMTP wired through environment values.                                                                           |
| Resource identity providers | Yours                                             | OIDC and OAuth 2.0 against your existing providers.                                                                                 |
| Observability               | Yours                                             | Prometheus metrics, OTLP export, and shipped alert rules consumed by your collectors and dashboards.                                |
| Object storage              | Yours, optional                                   | S3-compatible endpoint for audit Parquet export; disabled by default.                                                               |
| Networking and TLS          | Yours                                             | Loopback-only ports fronted by your proxy on VMs; Ingress and NetworkPolicy values on Kubernetes.                                   |
| Backups and recovery        | Shared                                            | Caracal ships backup and restore tooling and migration hooks; you own schedules, storage, and PITR.                                 |
| Compute                     | Yours                                             | Any Docker host or conformant Kubernetes 1.30+, provisioned with the OpenTofu modules or your own IaC.                              |
