Caracal ships a built-in Secret Store and a pluggable backend layer for user-entered credentials such as provider client secrets. The rest of the platform reads and writes secrets through one interface, so switching backends never changes application behavior.

## Storage Tiers

| Secret class | Where it lives |
| --- | --- |
| User-entered credentials (provider client secrets and API keys) | The configured backend, addressed by a stable reference such as `zones/{zone}/providers/{provider}/secretConfig`. |
| Machine-generated material (zone signing keys, connection tokens, sink secrets, operator plan credentials) | Always sealed envelopes in their owning Postgres tables, regardless of backend. |

Every value, in every backend, is sealed under `SECRET_STORE_KEK` before it leaves the service: a fresh data key encrypts the value with ChaCha20-Poly1305 and the KEK wraps that data key. External backends only ever hold ciphertext.

## Choose a Backend

Set `CARACAL_SECRET_BACKEND` on the API and STS services. The default is `builtin` and requires no configuration beyond `SECRET_STORE_KEK`.

| Value | Backend |
| --- | --- |
| `builtin` | Encrypted `secret_store` table in Caracal's Postgres. Default. |
| `vault` | HashiCorp Vault KV v2. |
| `infisical` | Infisical. |
| `azurekeyvault` | Azure Key Vault. |
| `awssecretsmanager` | AWS Secrets Manager. |
| `gcpsecretmanager` | Google Secret Manager. |
| `custom` | Any HTTPS service implementing the custom REST contract below. |

The API owns writes and deletes. STS holds a read-only view of the data plane and caches external reads for 60 seconds; the builtin backend reads Postgres directly and is not cached.

## Backend Variables

| Backend | Variables |
| --- | --- |
| `builtin` | None. |
| `vault` | `CARACAL_VAULT_ADDR`, `CARACAL_VAULT_TOKEN`; optional `CARACAL_VAULT_MOUNT` (default `secret`), `CARACAL_VAULT_NAMESPACE`. |
| `infisical` | `CARACAL_INFISICAL_TOKEN`, `CARACAL_INFISICAL_PROJECT_ID`; optional `CARACAL_INFISICAL_URL` (default `https://app.infisical.com`), `CARACAL_INFISICAL_ENV` (default `prod`), `CARACAL_INFISICAL_PATH` (default `/`). |
| `azurekeyvault` | `CARACAL_AZURE_VAULT_URL`; either `CARACAL_AZURE_TENANT_ID`, `CARACAL_AZURE_CLIENT_ID`, and `CARACAL_AZURE_CLIENT_SECRET`, or none of them to use the VM or pod managed identity. |
| `awssecretsmanager` | `CARACAL_AWS_REGION` (or `AWS_REGION`); either `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` (optional `AWS_SESSION_TOKEN`), or neither to use the container credentials endpoint. |
| `gcpsecretmanager` | `CARACAL_GCP_PROJECT`; either `CARACAL_GCP_CREDENTIALS_FILE` (or `GOOGLE_APPLICATION_CREDENTIALS`), or neither to use the metadata server identity. |
| `custom` | `CARACAL_CUSTOM_SECRETS_URL`, `CARACAL_CUSTOM_SECRETS_TOKEN`. |

Token-valued variables support `*_FILE` variants on both services.

Operator-supplied backend endpoints (`CARACAL_VAULT_ADDR`, `CARACAL_INFISICAL_URL`, `CARACAL_AZURE_VAULT_URL`, `CARACAL_CUSTOM_SECRETS_URL`) must be `https` URLs; plain `http` is accepted only for loopback hosts, which keeps local development against a same-host backend working.

String-valued stores receive base64-encoded envelope bytes; the byte-oriented custom contract and the builtin store carry raw envelope bytes.

## Ambient Cloud Credentials

On managed platforms the cloud backends authenticate without long-lived static keys. When the static credential variables are absent, each backend resolves its identity from the platform:

| Backend | Ambient source |
| --- | --- |
| `awssecretsmanager` | The container credentials endpoint (`AWS_CONTAINER_CREDENTIALS_FULL_URI` or `AWS_CONTAINER_CREDENTIALS_RELATIVE_URI`), as injected by EKS Pod Identity and ECS task roles; `AWS_CONTAINER_AUTHORIZATION_TOKEN` and `AWS_CONTAINER_AUTHORIZATION_TOKEN_FILE` are honored. |
| `azurekeyvault` | The IMDS managed-identity endpoint of the VM or pod. |
| `gcpsecretmanager` | The metadata server service account, as configured by GKE Workload Identity. |

EC2 instance-profile credentials without a container endpoint and Vault auth methods beyond static tokens are not resolved ambiently; supply static credentials or a token with a sufficient TTL and renew it through your secret delivery pipeline.

## Custom REST Contract

A custom backend is any HTTPS service exposing three routes under a base URL, authenticated with a bearer token:

| Operation | Route | Behavior |
| --- | --- | --- |
| Write | `PUT {base}/secrets/{ref}` | Stores the `application/octet-stream` request body under `ref`. |
| Read | `GET {base}/secrets/{ref}` | Returns the stored bytes, or `404` when absent. |
| Delete | `DELETE {base}/secrets/{ref}` | Removes the value; `404` is treated as success. |

Requests carry `Authorization: Bearer <CARACAL_CUSTOM_SECRETS_TOKEN>` and time out after 10 seconds. Stored payloads are sealed envelopes, so a compromised custom store exposes ciphertext only.

## Failure Behavior

When an external backend is unreachable during provider creation or update, the API rolls the provider row back inside the same transaction and returns `secret_backend_unavailable`, so the stored configuration and the credential document never diverge. The STS caches external reads for 60 seconds and serves a stale entry for up to 10 minutes while the backend errors, so a short backend outage does not stop token exchange; Caracal never falls back to another backend. Backend traffic is visible as `caracal_secret_backend_operations_total` and `caracal_secret_backend_errors_total` on the API and `caracal_sts_secret_backend_reads_total` and `caracal_sts_secret_backend_errors_total` on the STS.

## Backend Housekeeping

| Backend | Behavior to know |
| --- | --- |
| `awssecretsmanager` | Deletes use `ForceDeleteWithoutRecovery`; rely on Caracal's own storage tiers rather than the Secrets Manager recovery window. |
| `gcpsecretmanager` | Each write adds a version and then destroys the versions before it, so exactly one live version exists and version storage does not accumulate. |
| `infisical` | Refs map to dotted secret names (`zones.{zone}.providers.{provider}.secretConfig`) inside the configured project, environment, and path. |

## Rotate the Master Key

Envelopes embed a fingerprint of the key that sealed them, so reads route to the right key during a rotation window:

1. Deploy the new key as `SECRET_STORE_KEK` and the retiring key as `SECRET_STORE_KEK_PREVIOUS` (both support `*_FILE`) on the API and STS, then restart them. Writes seal under the new key immediately; existing envelopes keep decrypting under the previous key. In the Helm chart, set `secrets.kekRotation=true` and provide the `secretStoreKekPrevious` key in the runtime secret.
2. Re-seal everything under the new key from the API environment: `node scripts/rotate-secret-store-kek.mjs`. The sweep covers the `secret_store` table, machine-tier envelopes, and external backend documents; it is idempotent and safe alongside live traffic.
3. Remove `SECRET_STORE_KEK_PREVIOUS` from every service environment and restart.

## Migrate Between Backends

Stored documents are envelopes bound to backend-independent references, so a migration copies ciphertext without ever touching the master key:

1. Keep `CARACAL_SECRET_BACKEND` pointing at the current backend and export the target backend's connection variables.
2. Run `node scripts/migrate-secret-backend.mjs <target-kind>` from the API environment. Re-running is safe; writes are idempotent upserts.
3. Switch `CARACAL_SECRET_BACKEND` to the target on the API and STS and restart them. The STS picks up relocated documents within its 60-second cache window.
4. Delete the documents from the source backend once the platform is verified against the target.
