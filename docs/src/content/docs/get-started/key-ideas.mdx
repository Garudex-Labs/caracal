This page is the product mental model. It deliberately uses the terms operators and application developers need first, then maps them to the precise reference terms used deeper in the docs.

## The core promise

Run agents through Caracal, give them scoped access to tools and providers, and get enforceable audit and revocation without exposing provider secrets.

That promise becomes one default workflow:

1. Create a **zone** for the environment or tenant.
2. Register an **agent app**.
3. Register a **tool/provider resource** with scopes and an upstream URL.
4. Choose a **policy template** and activate it.
5. Run the agent through the SDK or `caracal run`.
6. Send tool/provider calls through the **Gateway**.
7. Use audit to see the authorization decision and the upstream action result.

Everything else is an advanced path.

## User-facing vocabulary

| User-facing term | Precise reference term | What it means |
| --- | --- | --- |
| Authority session | STS session / subject session | The root authority an agent run exchanges from. |
| Agent run | Agent session | The Coordinator-tracked execution of one agent instance. |
| Short-lived access token | Mandate | The signed Caracal token used for Gateway or direct-resource access. |
| Tool/provider resource | Resource identifier | The target an agent requests access to. |
| Provider route | Gateway binding | The Gateway routing entry created from a resource with an upstream URL. |
| Delegated permission | Delegation edge | A narrower permission from one agent run to another. |
| Restriction | Caveat / constraint | Scope, TTL, hop, budget, or policy condition attached to delegated permission. |

Use the left column when explaining Caracal to most users. Use the right column when reading API, architecture, or policy reference pages.

## What each product object does

### Zone

A zone is the isolation boundary. It owns agent apps, resources, providers, policies, signing keys, and audit data. A token issued in one zone is not valid in another.

### Agent app

An agent app identifies the workload that will act. The app can start agent runs, request scoped access, and appear in audit records. The app secret authenticates the agent to Caracal; it is not a provider credential.

### Tool/provider resource

A resource describes something an agent may call: an MCP server, REST API, SaaS provider, database-facing service, webhook, queue, or internal tool. A resource declares:

- the scopes an agent can request;
- the upstream URL when the Gateway should proxy the call;
- the optional provider credential source the Gateway may broker.

When a resource has an upstream URL and Gateway app identity, Caracal creates or updates the provider route automatically. Users should not manage the internal routing table directly.

### Policy template

A policy template is the starting point for authorization. It decides whether the agent app, requested scopes, resource, agent run state, and delegated permission are allowed. New users should start with `baseline-scopes`, then add resource, delegation, or step-up templates as the integration matures.

Policies fail closed. If there is no active policy set, access is denied.

### Gateway

The Gateway is the default execution plane. It verifies Caracal access tokens, checks replay and revocation, resolves the provider route, brokers provider credentials inside the trusted boundary, forwards the upstream request, and emits action-result audit.

Agents should call providers through the Gateway unless they are using a certified direct verifier or adapter.

## Token and credential boundaries

There are three important credentials:

| Credential | Who should hold it | Purpose |
| --- | --- | --- |
| Agent app secret | Agent runtime or deployment secret store | Authenticates the app to Caracal. |
| Caracal access token | Agent runtime | Short-lived scoped proof that Caracal allowed the action. |
| Provider-native credential | Gateway or certified adapter only | Authenticates to the upstream provider. |

The public STS returns only Caracal access tokens. Provider-native credentials are resolved through the internal broker path for Gateway or certified adapters.

## Delegation in one rule

Delegation has one invariant:

```text
source = delegator
target = receiver
receiver uses the delegated permission
```

The source cannot use the receiver's delegated permission. Parenthood does not automatically grant access. Every delegated permission narrows authority through explicit resource, scope, TTL, hop, budget, and policy restrictions.

## Revocation in one rule

A Caracal access token is invalid when any of its authority anchors is revoked:

- root authority session;
- agent run;
- delegated permission;
- grant;
- zone or application emergency disablement;
- emergency key rotation.

Gateway and direct verifiers use the same contract: check the token signature, replay state, expiry, and revocation anchors before accepting the request. Gateway also re-checks revocation while streaming responses.

## Audit in one rule

Authorization audit answers **why access was allowed or denied**. Action execution audit answers **what happened when the Gateway or adapter made the upstream call**.

Together, audit should let an operator answer:

- Which agent run made the request?
- Which resource and scopes were requested?
- Which policy allowed or denied it?
- Which provider route and auth mode were used?
- Did the upstream call succeed, fail, or stop because revocation arrived?

## Integration paths

| Path | Product status | Use it when |
| --- | --- | --- |
| Gateway proxy | Default | The tool/provider is HTTP, MCP-over-HTTP, REST, or another Gateway-routable upstream. |
| Direct verifier | Advanced | You own the service and can enforce Caracal verification, replay, revocation, and result audit there. |
| Adapter | Advanced | The provider uses gRPC, external MCP, queues, webhooks, or a custom transport. |
| Provider SDK wrapper | Advanced | A provider SDK hides transport and needs a certified wrapper to preserve Caracal enforcement. |

Start with the Gateway proxy. Move to direct verification or adapters only when the provider protocol requires it.
