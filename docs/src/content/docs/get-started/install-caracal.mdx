Running Caracal locally takes two pieces:

- **The `caracal` CLI** - a single executable that starts and stops the local stack and launches your programs under Caracal's control.
- **Docker** - the local stack runs as containers: Caracal's services, a PostgreSQL database, a Redis cache, and the web console, the browser interface where you will create and manage everything in the next page.

You install one binary; Docker provides the rest. No source checkout, package build, or separate database install is required.

## Prerequisites

- a supported Linux, macOS, or Windows machine;
- Docker 25 or later with Compose v2;
- `curl` or a browser for downloading a release;
- permission to install one executable on your `PATH`.

## Install the CLI

On Linux or macOS, the install script downloads the right release archive for your platform, verifies its checksum, and places `caracal` on your `PATH`:

```sh
curl -fsSL https://raw.githubusercontent.com/Garudex-Labs/caracal/main/install.sh | sh
```

Prefer to download and verify manually, on any platform? Each [release](https://github.com/Garudex-Labs/caracal/releases) ships platform archives (`caracal-runtime-*`), a `manifest.json`, and a `SHA256SUMS` file.

<Tabs syncKey="os">
<TabItem label="Linux / macOS">

```sh
gh release download vX.Y.Z \
  --repo Garudex-Labs/caracal \
  --pattern 'caracal-runtime-linux-amd64-*.tar.gz' \
  --pattern SHA256SUMS
sha256sum --check SHA256SUMS   # macOS: shasum -a 256 --check SHA256SUMS
tar -xzf caracal-runtime-linux-amd64-vX.Y.Z.tar.gz
install -m 0755 caracal ~/.local/bin/caracal
```

Choose the `linux-arm64`, `darwin-amd64`, or `darwin-arm64` archive when that matches your machine, and make sure the destination directory is on your `PATH`.

</TabItem>
<TabItem label="Windows">

```powershell
gh release download vX.Y.Z `
  --repo Garudex-Labs/caracal `
  --pattern 'caracal-runtime-windows-amd64-*.zip' `
  --pattern SHA256SUMS
Get-FileHash .\caracal-runtime-windows-amd64-vX.Y.Z.zip -Algorithm SHA256
Expand-Archive .\caracal-runtime-windows-amd64-vX.Y.Z.zip -DestinationPath $env:LOCALAPPDATA\Programs\caracal
```

Compare the PowerShell hash with the matching entry in `SHA256SUMS`, then add `%LOCALAPPDATA%\Programs\caracal` to the user `Path` and open a new shell.

</TabItem>
</Tabs>

For deeper supply-chain checks, signatures, and container image verification, see [Verify a Release](/security/verify-releases/).

## Verify the Installation

The same commands work in every shell:

```sh
caracal --version
caracal status --help
```

Both commands should print without errors.

## Verify Docker

```sh
docker compose version
```

If this fails, install or start Docker Desktop (macOS, Windows) or Docker Engine with the Compose plugin (Linux) before continuing.

## Start the Local Stack

```sh
caracal up
caracal status --ready
```

`caracal up` pulls and starts the containers; `caracal status --ready` waits until every service reports healthy. When readiness succeeds, the web console is being served at [http://localhost:3001](http://localhost:3001).

:::note
Do not sign in yet. The console keeps registration closed by default, so the next page first allows your email from the terminal and configures a sign-in method, then walks you through everything in the browser.
:::

## Common Mistakes

- If the shell cannot find `caracal` after installation, reopen the shell or add the install directory to `PATH`.
- Do not use `caracal purge` to recover from a normal startup error; it intentionally removes all local state.
- If you have a Caracal source checkout for contribution work, keep it separate: this path uses the released `caracal` executable, not the source-tree `pnpm caracal` command.

## Expected Outcome

`caracal status --ready` exits successfully and [http://localhost:3001](http://localhost:3001) responds. Readiness proves the local services are up; nothing is protected yet - that is the next page.

## Platform Notes

| Platform | Architectures | Notes |
| --- | --- | --- |
| Linux | `amd64`, `arm64` | Requires `curl` or `wget`, `tar`, and `sha256sum` or `shasum`. Archive checksums are always verified; installing GitHub CLI `gh` additionally verifies provenance attestations. |
| macOS | `amd64`, `arm64` | GitHub CLI `gh` enables provenance verification; if Gatekeeper quarantines the binary, remove quarantine from the installed file. |
| Windows | `amd64` | GitHub CLI `gh` enables provenance verification; open a new shell after updating the user `Path`. |
