This page walks through integrating an agent with Caracal using the TypeScript, Python, or Go SDK. By the end you will have an agent run that calls a tool/provider resource through the Gateway and leaves both authorization and action-result audit behind.

The instructions assume you have a running stack and a generated runtime profile from the [Quickstart](./quickstart/).

## How the SDK works

The SDK wraps token exchange and agent lifecycle behind three primitives:

- **`spawn()`** opens an agent run, gets a short-lived Caracal access token, and runs your code inside that identity context.
- **`headers()`** returns the HTTP headers that propagate the agent's identity to downstream services (Authorization, traceparent, baggage).
- **`transport()`** wraps `fetch` (TS/Python) or `*http.Client` (Go) to inject those headers on Gateway-bound outbound requests.

When `spawn()` exits (normally or on error) the agent run is terminated in the Coordinator and active authority for that run is revoked.

## Configuration

All three SDKs read the same generated runtime profile. Use the profile from Console guided setup, or set `CARACAL_CONFIG` to the profile path provided by your platform team. The profile contains the zone, application, STS, Coordinator, secret-file path, and resource credentials needed for SDK-managed token exchange.

The default profile path is `$XDG_CONFIG_HOME/caracal/caracal.toml`, otherwise `~/.config/caracal/caracal.toml`. Environment variables remain available for platform-managed workloads, but the beginner path should start with the generated profile so local and deployed behavior use the same contract.

## Using `caracal run` with SDK examples

`caracal run` reads validated runtime config from platform environment variables or an explicit runtime profile, exchanges credentials with the STS, and injects tokens under the configured environment variable names.

Use `caracal run` for scripts that read the token environment variable generated in the Console runtime profile. For SDK processes, prefer `transport()` with either a configured upstream binding or the explicit Gateway helper. The helper returns the Gateway URL and `X-Caracal-Resource` header; `transport()` still injects `Authorization`, `traceparent`, and `baggage` from the active Caracal context.

For SDK processes, use `Caracal.connect()` with the same generated profile. See the language-specific SDK pages for complete environment setup.

Set `CARACAL_RESOURCE_ID` to the resource identifier from guided setup and `CARACAL_RESOURCE_PATH` to the real path on that protected resource that this integration should call.

## TypeScript

Install the SDK:

```sh
npm install @caracalai/sdk
```

Minimal example: spawn an agent run and call the real resource path you configured in guided setup.

```typescript
import { Caracal, AgentKind } from "@caracalai/sdk";

const caracal = Caracal.connect();
const resourceId = process.env.CARACAL_RESOURCE_ID;
const resourcePath = process.env.CARACAL_RESOURCE_PATH;
if (!resourceId) throw new Error("CARACAL_RESOURCE_ID is required");
if (!resourcePath) throw new Error("CARACAL_RESOURCE_PATH is required");

await caracal.spawn(async () => {
  const ctx = caracal.current();
  console.log("Agent session:", ctx?.agentSessionId);

  const request = caracal.gatewayRequest(resourceId, resourcePath);
  const client = caracal.transport();
  const response = await client(request.url, {
    method: "POST",
    headers: { ...request.headers, "Content-Type": "application/json" },
    body: JSON.stringify({}),
  });

  const result = await response.json();
  console.log(result);
}, { kind: AgentKind.Ephemeral });
```

`Caracal.connect()` auto-detects the generated profile first, then falls back to environment variables. See [Configuration precedence](/reference/config-precedence/) for the full resolution order. To protect an inbound Express route, use the verifier connector in [Protect an Express App](../../guides/protect-express/).

## Python

Install the SDK:

```sh
pip install caracalai-sdk
```

Minimal example using `async with` spawn:

```python
import asyncio
import os
from caracalai_sdk import Caracal, AgentKind

caracal = Caracal.connect()
resource_id = os.environ["CARACAL_RESOURCE_ID"]
resource_path = os.environ["CARACAL_RESOURCE_PATH"]

async def main():
    async with caracal.spawn(kind=AgentKind.EPHEMERAL) as ctx:
        print("Agent session:", ctx.agent_session_id)

        # transport() returns an httpx.AsyncClient pre-configured with Caracal access headers.
        request = caracal.gateway_request(resource_id, resource_path)
        async with caracal.transport() as client:
            response = await client.post(
                request.url,
                headers=request.headers,
                json={},
            )
            print(response.json())

asyncio.run(main())
```

To protect an inbound Python service, route it through Gateway first or use a verifier connector. Context-only SDK middleware is documented in the SDK reference for services that already sit behind a verified boundary.

## Go

Install the SDK:

```sh
go get github.com/garudex-labs/caracal/packages/sdk/go
```

Minimal example:

```go
package main

import (
    "context"
    "fmt"
    "io"
    "net/http"
    "os"

    caracal "github.com/garudex-labs/caracal/packages/sdk/go"
)

func main() {
    c, err := caracal.Connect()
    if err != nil {
        panic(err)
    }
    resourceID := os.Getenv("CARACAL_RESOURCE_ID")
    resourcePath := os.Getenv("CARACAL_RESOURCE_PATH")
    if resourceID == "" || resourcePath == "" {
        panic("CARACAL_RESOURCE_ID and CARACAL_RESOURCE_PATH are required")
    }

    err = c.Spawn(context.Background(), func(ctx context.Context) error {
        cc, _ := c.Current(ctx)
        fmt.Println("Agent session:", cc.AgentSessionID)

        target, err := c.GatewayRequest(resourceID, resourcePath)
        if err != nil {
            return err
        }
        req, err := http.NewRequestWithContext(ctx, http.MethodPost, target.URL, nil)
        if err != nil {
            return err
        }
        req.Header = target.Header.Clone()
        req.Header.Set("Content-Type", "application/json")
        resp, err := c.Transport(nil).Do(req)
        if err != nil {
            return err
        }
        defer resp.Body.Close()
        body, _ := io.ReadAll(resp.Body)
        fmt.Println(string(body))
        return nil
    })
    if err != nil {
        panic(err)
    }
}
```

### Protect a net/http service

```sh
go get github.com/garudex-labs/caracal/packages/connectors/nethttp/go
```

```go
import (
    caracal "github.com/garudex-labs/caracal/packages/sdk/go"
    "github.com/garudex-labs/caracal/packages/connectors/nethttp/go"
)

c, _ := caracal.Connect()

mux := http.NewServeMux()
mux.HandleFunc("/tool", func(w http.ResponseWriter, r *http.Request) {
    ctx, _ := c.Current(r.Context())
    fmt.Fprintln(w, "Agent session:", ctx.AgentSessionID)
})

http.ListenAndServe(":8090", c.ContextMiddleware(mux))
```

For direct inbound protection, use the net/http verifier connector before binding SDK context.

## Delegation example

An agent can grant narrower delegated permission to another agent. The parent/source creates the permission, the child/target receives it, and only the receiver can use it.

```typescript
// TypeScript example
await caracal.spawn(async () => {
  // Delegate to a child agent session with narrowed scope.
  await caracal.delegate(
    {
      to: childAgentSessionId,
      toApplicationId: "app2",
      scopes: ["resource://example:read"],
      constraints: { ttlSeconds: 300 },
    },
    async () => {
      // Within this block, the delegated context is active.
      const child = caracal.current();
      console.log("Delegation edge:", child?.delegationEdgeId);
    }
  );
});
```

Revoking the delegated permission from the Console or admin SDK terminates the child agent run and all descendants immediately.

## What happens if the policy denies the exchange

If policy denies access, the STS returns HTTP 403 and the SDK throws an error. No Caracal access token is issued. The denial is recorded in the audit ledger with the determining policies and diagnostics that caused it. Use the Console Audit view or `the Console` to inspect the decision.

## Prefer Gateway before direct verification

The Gateway path is the product default because it keeps provider secrets out of agent code, centralizes revocation, and records action-result audit. Use SDK middleware or connector packages only when the resource is yours and can enforce signature checks, replay protection, revocation, and result audit itself. Use adapter paths for provider SDKs or protocols the Gateway cannot proxy only when the adapter preserves the same enforcement contract.
