Caracal lets you run agents with scoped access to tools and providers without putting provider secrets in agent code. This page explains the default product path and where the advanced paths fit.

## The core problem

AI agents and autonomous workflows need access to tools, APIs, and data. Giving them long-lived credentials causes two problems: you cannot know what the agent will do with them before it acts, and you cannot revoke access granularly once something goes wrong.

Caracal addresses both problems by giving the agent a short-lived, scoped Caracal access token only after policy allows the requested action. The token expires quickly, can be revoked by multiple authority anchors, and never contains provider-native credentials.

## The golden path

The default path is:

```text
agent app
  → Caracal SDK opens an agent run
  → agent calls the Gateway
  → public STS issues only Caracal access tokens
  → internal broker resolves provider credentials for the Gateway
  → Gateway forwards the upstream call
  → audit records the decision and action result
```

This is the path new integrations should use. It gives the strongest default security boundary: agents never see provider OAuth tokens or API keys, the Gateway checks revocation before and during requests, and every proxied action has an execution audit event.

## The enforcement point

The public STS issues short-lived Caracal access tokens. It does not return provider-native credentials to agent-facing callers. Before signing a token, it:

1. Receives an exchange request carrying the agent identity and requested tool/provider resource.
2. Loads the active policy set for the zone.
3. Evaluates OPA with the agent, resource, requested scopes, session state, delegation state, and trace context.
4. Signs a token only when policy returns `decision = "allow"` and `evaluation_status = "complete"`.
5. Emits an authorization audit event for every resource decision.

If no policy set is active in the zone, the STS installs a deny-all fallback — no agent can exchange credentials until a policy is activated. There is no default-allow state.

## The Gateway

When an agent calls a tool, MCP server, REST API, or provider endpoint through the Gateway, the Gateway:

1. Verifies the Caracal token signature against the zone key.
2. Checks replay and revocation anchors.
3. Resolves the provider route for the requested resource.
4. Calls the internal broker path when provider credentials are needed.
5. Substitutes the provider OAuth token or API key only on the upstream request.
6. Emits an action execution audit event with status, latency, result class, auth mode, and revocation interruption state.

The agent presents a Caracal access token; the Gateway handles the credential the upstream expects.

## Delegation

An agent can grant a narrower delegated permission to another agent. The invariant is:

```text
source = delegator
target = receiver
receiver uses the delegated permission
```

Parenthood alone does not grant access. Each delegated permission narrows authority through explicit scope, resource, TTL, hop, and policy constraints. The source cannot use the receiver's delegated permission, unrelated agents cannot use it, and revocation terminates downstream authority.

## The audit ledger

Every authorization decision produces an append-only audit event. Gateway-proxied calls also produce action execution audit events. Together they answer:

- Was the call allowed or denied?
- Which policy and template version determined the outcome?
- Which agent run and delegated permission were involved?
- Which provider route and auth mode were used?
- Did the upstream call succeed, fail, or stop because revocation arrived?

Nothing is ever updated or deleted in the audit table. Query the trail through the TUI and CLI (`audit tail`, `explain`).

## What this is not

Caracal does not replace your LLM framework, agent scheduler, or prompt router. It sits alongside those systems as the access-control and audit boundary. Caracal does not inspect or filter tool-call content; it enforces whether the call may happen, which credential is used, how it is audited, and how quickly it can be revoked.

## Advanced paths

Use these only when the Gateway path does not fit:

| Path | When to use |
| --- | --- |
| Direct verification | You own the resource and can verify Caracal tokens, replay, and revocation in that service. |
| External MCP adapter | The MCP server cannot be modified but can sit behind a trusted adapter. |
| Custom gRPC adapter | The provider uses a protocol the Gateway cannot proxy safely. |
| Provider SDK wrapper | A provider SDK hides transport and needs a certified wrapper to preserve Caracal headers, revocation, and action audit. |
