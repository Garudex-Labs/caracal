Caracal is a self-hosted execution boundary for AI agents. Run agents through Caracal, give them scoped access to tools and providers, and get enforceable audit and revocation without exposing provider secrets to agent code.

The default product path is intentionally narrow:

1. Create a zone.
2. Register an agent app.
3. Register a protected resource with scopes and an upstream URL when Gateway routing is needed.
4. Activate an access policy.
5. Run the workload through the SDK or `caracal run`.
6. Send protected calls through the Gateway.
7. Explain the first audit decision and action result.

Other provider flows fit into three buckets: connector-verified services you own, application-managed external provider calls that keep Caracal context for attribution, and future native broker plugins. Start with the Gateway path unless you know which bucket applies.

For a first integration, keep the mental model simple: use Console guided setup to create the zone, agent app, protected resource, policy, and runtime profile; use the SDK or `caracal run` to run the workload; send protected calls through Gateway. SDK middleware is for propagating already-verified context, not for replacing the Gateway enforcement boundary.

| Platform team | [Installation](./installation/) → [CLI and Console](/runtime-console/cli-and-console/) → [Configuration File](/runtime-console/config-file/) | Treating `caracal.toml` as product state; it is runtime input for workloads and SDKs. |

## Prerequisites

- **Docker** 24+ with the Compose v2 plugin
- **curl** or **wget** (required by the shell install script)
- A POSIX shell (Linux or macOS) or PowerShell 7+ (Windows)

SDK requirements depend on language. See [Installation](./installation/) for details.
