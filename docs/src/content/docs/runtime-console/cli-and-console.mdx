Use this page before reaching for a command. Caracal separates local runtime lifecycle from product management so workload execution does not depend on admin tokens or selected zones.

## Ownership Rule

```text
local lifecycle and process execution -> caracal
human product management              -> web console
scripted product management           -> Control API or Admin SDK
```

The top-level runtime CLI intentionally contains only `up`, `down`, `status`, `upgrade`, `purge`, `run`, and `web`.

:::note[FAQ]
[Why are zone and policy commands in the web console instead of the `caracal` CLI?](/reference/faq/#faq-015)
:::

## Runtime CLI Tasks

| Outcome | Command |
| --- | --- |
| Start the local stack | `caracal up` |
| Stop the local stack | `caracal down` |
| Stop and remove volumes | `caracal down -v` |
| Probe service health | `caracal status` |
| Probe dependency readiness | `caracal status --ready` |
| Emit structured status | `caracal status --json` |
| Upgrade local runtime assets | `caracal upgrade` |
| Remove local stack/runtime state | `caracal purge` |
| Run a process with resource tokens | `caracal run -- <command>` |
| Open the packaged local web console | `http://localhost:3001` after `caracal up` |
| Launch the local web development server | `caracal web` |

Runtime CLI commands must not require admin tokens, selected zones, product credentials, or Control credentials.

## Web Console Tasks

| Outcome | Web console section |
| --- | --- |
| Create or select zones | **Zones** |
| Register confidential agent applications | **Applications** |
| Configure provider credential sources | **Providers** |
| Register protected resources and scopes | **Resources** |
| Author access policies | **Policies** |
| Simulate and activate policy sets | **Policy Sets** |
| Inspect authority sessions | **Sessions** |
| Enable and manage Control API access | **Control** |
| Search audit records and trace selected requests | **Audit** |
| Trace one known request ID | **Audit** |
| Inspect or manage agent sessions | **Agents** |
| Inspect or revoke delegation edges | **Delegation** |
| Run health, readiness, zone, and preflight checks | **Diagnostics** |

Product-management workflows are not mirrored as top-level runtime CLI commands. Keeping them in the web console prevents command drift and keeps human operations tied to the same engine/admin helpers.

## Automation Tasks

Use the Admin SDK from trusted operator environments when scripts need to create zones. Use the Control API for zone-bound automation over applications, resources, policies, authority sessions, agents, delegation edges, audit queries, and Control credentials. Enable Control from the web console's **Control** section, then call the authenticated HTTP surface from CI or automation.

Automation must run with operator credentials only in trusted control-plane environments. Agent workloads should use `caracal run` or application credentials and must not receive `CARACAL_ADMIN_TOKEN`, `CARACAL_COORDINATOR_TOKEN`, `CARACAL_SECRETS_DIR`, or mounted operator secret files.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| The web console cannot reach product data | Run `caracal up` and confirm `caracal status --ready`; for the development launcher, use `caracal web --allow-offline` only for sign-in or UI-only checks. |
| A command for zones, policies, or audit is missing from `caracal` | Open the web console. Those workflows are intentionally web-console-owned. |
| Product management is needed in CI | Use the Control API or Admin SDK with scoped operator credentials. |
| Runtime command asks for product-management context | Treat it as a bug; runtime lifecycle must not depend on admin or zone state. |
