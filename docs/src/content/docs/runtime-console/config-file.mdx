Workload configuration has two consumers. `caracal run` needs only a workload identity; its credential bindings live in the web console. SDK credential loaders read a local profile. Neither is Console state and neither creates, selects, or manages zones.

## Workload Identity for caracal run

`caracal run` authenticates as one workload, created on the console's **Launcher** page, and fetches that workload's launch bindings from STS. The environment carries only identity:

| Variable                       | Meaning                                               |
| ------------------------------ | ----------------------------------------------------- |
| `CARACAL_WORKLOAD_ID`          | Workload identifier from the Launcher page. Required. |
| `CARACAL_WORKLOAD_SECRET`      | Inline local-development workload secret.             |
| `CARACAL_WORKLOAD_SECRET_FILE` | Cloud/custom path to a mounted workload-secret file.  |
| `CARACAL_STS_URL`              | Cloud/custom STS URL override for credential minting. |

Set only one of `CARACAL_WORKLOAD_SECRET` and `CARACAL_WORKLOAD_SECRET_FILE`. Local dev and stable service URLs resolve automatically; set `CARACAL_STS_URL` only for cloud deployments, containers, or custom service locations.

When neither secret variable is set, local dev and stable launches read the owner-only secret file at:

```text
<Caracal config dir>/runtime/<workload_id>/secret
```

The OS Caracal config directory is `$XDG_CONFIG_HOME/caracal` (or `~/.config/caracal`) on Linux, `~/Library/Application Support/Caracal` on macOS, and `%APPDATA%\Caracal` on Windows. Set `CARACAL_CONFIG_HOME` to override it on any platform.

You create the secret file yourself. The web console holds the workload secret sealed in the Secret Store - reveal it from the Launcher page whenever you provision - but never writes to your machine:

```bash
mkdir -p ~/.config/caracal/runtime/<workload_id>
printf '%s' '<workload secret>' > ~/.config/caracal/runtime/<workload_id>/secret
chmod 600 ~/.config/caracal/runtime/<workload_id>/secret
```

The default path is read only in local dev and stable modes; production launches (`CARACAL_ENV=production`) require explicit secret configuration.

## Launch Bindings

Which environment variables `caracal run` injects, which resources fill them, the scopes each credential carries, and optional-credential behavior are all part of the workload's launch bindings, authored on the **Launcher** page in the web console. STS serves the bindings to the workload after it proves possession of the workload secret, so the zone, resource identifiers, and variable names never live on the machine that launches the workload.

Every workload is scoped to its zone, and each save is attributed: the Launcher page shows which console operator configured the bindings and when, and the change appears in the zone's admin audit feed under that operator's profile.

Each binding selects the scopes its credential should carry; leaving all scopes unselected requests the resource's full scope set. The Launcher page also renders the exact launch commands for the workload. See [Run Workloads](/runtime-console/runtime/) for the launch flow.

## SDK Profile

SDK credential loaders resolve a local profile instead of calling STS for a manifest:

| Order | Source                                                                                                     |
| ----- | ---------------------------------------------------------------------------------------------------------- |
| 1     | `CARACAL_CONFIG`, when set and the file exists.                                                            |
| 2     | Environment runtime config (`CARACAL_ZONE_ID`, `CARACAL_APPLICATION_ID`, secret and credential variables). |

SDKs do not search the current working directory, home directory, or OS config directory. Secret and credential files are read only when an explicit profile field or `*_FILE` environment variable names them.

### TOML Schema

```toml
zone_id = "zone_prod"
application_id = "app_agent"
default_ttl_seconds = 900

[[credentials]]
resource = "resource://openai"
upstream_prefix = "https://api.openai.com/v1"

[[optional_credentials]]
resource = "resource://mandate-aware-api"
```

### Field Reference

| Field                    | Meaning                                                                       |
| ------------------------ | ----------------------------------------------------------------------------- |
| `sts_url`                | Cloud/custom STS endpoint for token exchange.                                 |
| `gateway_url`            | Cloud/custom Gateway endpoint override for SDK transports.                    |
| `coordinator_url`        | Cloud/custom Coordinator endpoint override for SDK agent/delegation features. |
| `zone_id`                | Zone created in the web console.                                              |
| `application_id`         | Confidential application used by the workload.                                |
| `app_client_secret_file` | Cloud/custom secret-file path override.                                       |
| `app_client_secret`      | Inline local-development secret.                                              |
| `default_ttl_seconds`    | Default Session TTL used when a call does not provide one.                    |
| `credentials[]`          | Resource audiences and optional Gateway routing bindings.                     |
| `optional_credentials[]` | Additional resource audiences and optional Gateway routing bindings.          |

SDK credential entries use `resource` and optional `upstream_prefix`. Launcher binding fields such as `env`, `credential_type`, `on_failure`, and `continue_on_failure` belong to `caracal run` launch bindings and are not SDK profile behavior.

## Security Rules

- Store the workload secret at the runtime path with owner-only permissions for dev and stable launches.
- Use explicit secret-file paths for cloud deployments, containers, or external secret stores.
- Secret files and explicit config files must not be group- or world-writable.
- Non-local insecure `http://` STS URLs are rejected unless explicitly allowed by environment.
- Environment names that affect runtime loading, such as `NODE_OPTIONS`, `LD_PRELOAD`, and dynamic-library path variables, are blocked as binding targets.

## Troubleshooting

| Symptom                                      | Check                                                                             |
| -------------------------------------------- | --------------------------------------------------------------------------------- |
| `workload identity not found`                | Set `CARACAL_WORKLOAD_ID` and a workload secret source.                           |
| `no credential bindings configured`          | Define launch bindings for this workload on the Launcher page in the web console. |
| Provider key is not injected                 | Confirm the provider config allows runtime injection.                             |
| Permission error on the secret file          | The file must be owner-only; `chmod 600` it.                                      |
| `CARACAL_CONFIG` is set but ignored (SDKs)   | The file must exist; when it is set, no other profile path is considered.         |
| Optional launch credential stops the process | Set the binding's unavailable behavior to warn on the Launcher page.              |
