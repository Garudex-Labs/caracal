## Definition

A **Resource Binding** is a row in the `gateway_resource_bindings` table that maps a resource identifier (`rid`) to:

- **Upstream URL** — the target URL the gateway forwards the request to.
- **Auth mode** — how the gateway authenticates to the upstream (`bearer`, `provider_oauth`, `provider_apikey`, `none`).
- **Provider credential reference** — for provider auth modes, the vault reference to the encrypted credential.
- **Zone and application scope** — the zone and application this binding is valid for.

Resource bindings are the registration mechanism that makes a resource reachable through Caracal. Unregistered resources are not routable through the gateway — calls to them bypass Caracal entirely.

## Why it exists

The gateway needs to know two things before forwarding a request:

1. Where does this resource actually live (the upstream URL)?
2. What credential should be substituted in the `Authorization` header?

Resource bindings answer both questions in one database row. This separates the *caller's view* of a resource (a stable `rid`) from the *runtime routing reality* (a URL that might change, an API key that rotates).

For AI agent workloads, bindings also enable provider credential vaulting: an agent can call the OpenAI API through the gateway without holding an OpenAI API key. The gateway substitutes the vaulted credential from the ZEK-encrypted vault.

## Gateway routing flow

1. SDK sends request to `gatewayUrl` with `X-Caracal-Resource: <rid>` header and subject-token bearer.
2. Gateway resolves `(zone_id, application_id)` from the resource binding table (polled every 30 s).
3. Gateway exchanges the subject token with STS for a per-call credential. STS returns an `UpstreamDirective` containing the resolved upstream URL and the auth mode.
4. Gateway replaces `Authorization` with the per-call credential, strips `X-Caracal-*` headers, and forwards to the upstream URL.

## Auth modes

| Mode | Description |
|---|---|
| `bearer` | Gateway forwards the per-call bearer token STS minted for this exchange. |
| `provider_oauth` | Gateway substitutes a vaulted OAuth access token for a registered provider (e.g., OpenAI). Token refresh is handled transparently. |
| `provider_apikey` | Gateway substitutes a vaulted API key. The key never leaves the STS vault in plaintext. |
| `none` | No `Authorization` header is set. Used for public upstream endpoints. |

## Registering a resource binding

```bash title="Register a resource binding via the CLI"
caracal init resource \
  --rid openai-chat \
  --upstream https://api.openai.com/v1 \
  --auth-mode provider_apikey \
  --workspace my-workspace

# The SDK then routes calls to openai-chat through the gateway.
```

```typescript title="Call a registered resource through the gateway — TypeScript SDK"
import { CaracalClient } from '@caracalai/sdk'

const client = new CaracalClient({
  coordinatorUrl: process.env.CARACAL_COORDINATOR_URL,
  gatewayUrl: process.env.CARACAL_GATEWAY_URL,
})

const session = await client.run({
  intent: { action: 'api:call', resource: 'openai-chat' },
})

// The gateway resolves the binding, substitutes the vaulted API key,
// and forwards to https://api.openai.com/v1. The agent never sees the key.
const response = await session.fetch('openai-chat/chat/completions', {
  method: 'POST',
  body: JSON.stringify({ model: 'gpt-4o', messages: [...] }),
})
```

## Constraints and limits

- Resource bindings are polled by the gateway every 30 seconds. A newly registered binding takes up to 30 seconds to become active.
- Unregistered resources are not routable through the gateway. The SDK bypasses the gateway for unregistered calls.
- The `rid` is stable across URL changes. Update the upstream URL in the binding without changing application code.
- Each binding is scoped to `(zone_id, application_id)`. The same `rid` can have different upstream URLs for different applications.
