## Definition

Caracal enforces a rule: **no action runs without a prior signed mandate**.

A mandate is a short-lived signed JWT that binds three things together: *who* is acting ([Principal](/concepts/principal/)), *what rules govern the action* ([Policy](/concepts/policy/)), and *what the action is* ([Intent](/concepts/intent/)). Before any code executes against a resource, Caracal's gateway verifies that a valid mandate exists for exactly that action.

This is pre-execution enforcement — authority is checked and locked before the first byte is sent, not audited after the fact.

## Why it exists

AI agents running with ambient long-lived credentials are a security liability. A compromised credential is a permanent, unconstrained breach. An agent that can call any API with a bearer token it holds indefinitely has no blast radius control.

Caracal replaces ambient credentials with scoped, time-bounded mandates. Each mandate is valid for one declared action, expires by design, and can be revoked mid-session. The system treats authority as something that must be earned per-action, not inherited per-session.

## The three-primitive loop

Every operation Caracal enforces follows the same loop:

1. **Declare intent.** The SDK or application declares what action the principal wants to take: the action verb, the target resource, and any relevant context. This is the [Intent](/concepts/intent/).

2. **Issue mandate.** The STS receives the intent, authenticates the principal, evaluates the declared intent against the workspace's [Policy](/concepts/policy/), and — if allowed — issues a signed [Mandate](/concepts/mandate/) with a short TTL.

3. **Execute under authority.** The gateway receives the outbound request, verifies the mandate's signature, checks its intent matches the actual call, calls STS for a per-call access credential, and forwards the request. The [Ledger](/concepts/ledger/) records the decision.

If step 2 is denied — the policy rejects the intent — execution never starts. If the mandate expires or is revoked between steps 2 and 3, execution stops.

## Scope and boundaries

**What Caracal enforces:**
- API calls routed through the gateway to registered resources
- Provider credential access (Caracal substitutes vaulted credentials for registered providers)
- Agent-to-agent delegation (authority passed between agents is always scoped to what the delegator holds)

**What Caracal does not enforce:**
- Calls to unregistered upstreams (these bypass the gateway by operator choice)
- Internal memory, file system access, or in-process operations
- Calls between two agents that share the same principal and do not use delegation

**The gateway is the enforcement point.** Code that does not route through the gateway runs outside Caracal's guarantees. This is intentional: operators register only the resources they need controlled, and Caracal is explicit about what it covers.

## Constraints

- **Pre-execution only.** Caracal does not intercept in-flight or completed requests; it gates them before they start.
- **Gateway-enforced.** Only calls routed through the gateway receive enforcement. Unregistered calls are out of scope by operator choice.
- **No ambient authority.** The STS never issues long-lived tokens. Every mandate has a TTL; there is no "permanent access" mode.
- **Fail-closed.** JTI replay tracking defaults to fail-closed. If Redis is unreachable and `JTI_FAIL_OPEN=false`, requests are rejected rather than allowed through.
