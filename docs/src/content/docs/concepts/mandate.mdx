A mandate is the credential Caracal issues after policy allows an exchange. It is an ES256-signed JWT that asserts who may act, which zone it belongs to, what resource authority it carries, and which revocation anchors can cut it off.

## Two mandate uses

Every mandate carries a `use` claim. There are exactly two valid values:

### Session mandate (`use = "session"`)

A session mandate is issued at the start of an execution session when no prior mandate is presented to the STS. It is exchange-only authority: it proves the registered principal controls the execution session and may ask the STS for narrower resource mandates.

- **TTL**: 60 minutes
- **Purpose**: anchors an execution session and its revocation lineage
- **Re-use**: can be presented to STS as the RFC 8693 `subject_token`; cannot be accepted by Gateway or production resources as an execution credential

### Resource mandate (`use = "resource"`)

A resource mandate is issued when a session mandate is presented to the STS. It is narrowed to a concrete resource set, scope set, session state, delegation edge, and policy decision.

- **TTL**: 15 minutes
- **Purpose**: authorizes Gateway-mediated HTTP or connector-verified execution
- **Re-use**: cannot be presented as `subject_token` for further exchange, which prevents a resource-scoped credential from being used to mint authority for another resource

The distinction is mechanical: session mandates can only be exchanged at STS, and resource mandates can only be enforced at Gateway or connectors. Mixing those roles is a security error.

## JWT claims

A mandate carries the following claims in addition to standard JWT registered claims (`iss`, `sub`, `aud`, `exp`, `iat`, `jti`):

| Claim | Type | Description |
| ----- | ---- | ----------- |
| `use` | string | `"session"` or `"resource"` |
| `zone_id` | string | Zone this token is bound to |
| `client_id` | string | Application ID of the issuing principal |
| `scope` | string | Space-separated OAuth scopes granted |
| `target` | string[] | Resource identifiers this token covers; required for resource mandates |
| `sid` | string | STS execution session ID for this mandate |
| `root_sid` | string | Root execution session ID inherited by derived resource mandates |
| `agent_session_id` | string | Coordinator agent session ID |
| `delegation_edge_id` | string | ID of the delegation edge, if delegated |
| `source_session_id` | string | Source agent session ID in a delegation edge |
| `target_session_id` | string | Target agent session ID in a delegation edge |
| `delegation_path` | string[] | Ordered list of delegation path IDs |
| `delegation_chain` | object[] | Full chain of `{application_id, agent_session_id, delegation_edge_id}` hops |
| `hop_count` | number | Number of delegation hops from root |
| `delegation_graph_epoch` | number | Epoch of the delegation graph at issuance |

The `target` claim on resource mandates prevents scope creep: the Gateway verifies that the resource being accessed appears in `target`, and rejects mandates that cover a different resource even if they have matching scopes.

The `root_sid` claim keeps the authority graph simple: session mandates set `root_sid` to their own `sid`, and resource mandates inherit the same value. Revoking the root execution session lets Gateway and connectors reject every derived mandate even when each resource mandate has its own `sid`.

## Signing and verification

Mandates are signed with ES256 (ECDSA P-256 + SHA-256). Each zone has its own signing key pair.

**Issuance**: The STS decrypts the zone's signing key from `dek_ciphertext` in the database using the `ZONE_KEK`, then signs the JWT with the zone's private key.

**Verification**: The Gateway and connectors retrieve the zone's public keys from the STS's JWKS endpoint:

```
GET /zones/{zoneId}/.well-known/jwks.json
```

The JWKS endpoint serves the two most recent signing keys for the zone, enabling key rotation without breaking in-flight tokens.

JWKS responses are cached in-process. The Gateway caches per zone and refreshes every 5 minutes.

## JTI replay protection

Each mandate carries a unique `jti` (JWT ID). Resource mandates are subject to replay protection:

1. On issuance, the STS writes `audit:jti:{jti}` to Redis using `SETNX` with TTL equal to the token lifetime.
2. If `SETNX` fails, the JTI already exists — the STS rejects the exchange and emits a `jti_collision` audit event. Under normal operation this never occurs; it indicates clock skew or a UUID generation fault.
3. The Gateway maintains a parallel check using `seen:jti:{jti}` in Redis, also with `SETNX`. The first request carrying a resource mandate succeeds; any subsequent request with the same JTI is rejected with HTTP 401.

Session mandates are not Gateway execution credentials. They are reusable only as STS exchange input.

## Exchange, caching, and replay

The term "resource" describes the authority boundary, not a guarantee that every SDK method invocation always calls STS. The safe behavior depends on where enforcement happens:

| Path | Exchange behavior | Replay behavior |
| ---- | ----------------- | --------------- |
| Gateway-mediated HTTP | Gateway performs an STS exchange for each inbound request and forwards only after replay and revocation checks pass | Gateway rejects a repeated resource mandate JTI before the upstream sees it |
| Connector-verified call | SDK may cache a token for the same execution session, agent session, delegation edge, resource, and scope context until the token expires | The connector or resource-side verifier must enforce replay/revocation if Gateway is not in the path |
| Session-to-resource exchange | SDK re-presents the session mandate to STS to mint narrower resource authority | Session mandates are not accepted by Gateway as upstream authorization |

This is why provider and tool integrations are safest when routed through the Gateway: Caracal owns credential brokering, replay detection, revocation checks, and action audit in one enforcement point.

## Long-running execution

Authority must stay valid for the whole execution, not only at issuance.

| Situation | Required behavior |
| --- | --- |
| Gateway HTTP request | Token expiry is checked before forwarding. The upstream request may finish if authority stays valid; Gateway aborts streaming copies when revocation arrives. |
| Retry | A retry is a new action attempt. It must present a fresh Gateway request or a connector token whose replay contract allows the retry. |
| Long stream | The stream keeps the original request ID and trace. Gateway or adapter must re-check revocation during the stream and record whether revocation interrupted it. |
| Background work | The worker must receive the Caracal envelope in message headers, re-bind the authority context, reject expired or revoked authority before execution, and emit action-result audit. |
| Provider-side sub-action | Gateway or adapter must bound the operation it invokes. Provider-internal sub-actions are not individually authorized by Caracal unless the adapter instruments them as separate Caracal actions. |
| Agent termination | Termination, stale service heartbeat, delegated-permission revocation, grant revocation, or root authority revocation invalidates further use through the shared revocation contract. |

## Mandate lifetime and the exchange chain

```
Agent credentials → STS → session mandate (60 min)
                              │
                              └─ re-present as subject_token
                              │
                              ▼
                           STS → resource mandate (15 min) covering [resource://payments]
                              │
                              └─ present to Gateway → upstream
```

The session mandate is the exchange credential. It is minted at session start and stored in the agent's execution environment (injected by `caracal run` as `CARACAL_SUBJECT_TOKEN` for RFC 8693 compatibility). The SDK exchanges this session mandate for resource mandates as needed for outbound calls. Resource mandate STS sessions are linked to their parent session, and mandates carry `root_sid` for deterministic descendant revocation. Delegated resource mandates are capped by the shortest effective authority bound: STS TTL limit, delegation edge expiry, and any typed `ttl_seconds` constraint on the edge.

## Token response shape

The STS token exchange response:

```json
{
  "access_token": "<resource mandate JWT>",
  "token_type": "Bearer",
  "expires_in": 900,
  "scope": "read",
  "issued_token_type": "urn:ietf:params:oauth:token-type:access_token",
  "target_resources": ["resource://payments"],
  "upstreams": {
    "resource://payments": {
      "url": "https://payments.internal/api",
      "auth_mode": "caracal_jwt"
    }
  }
}
```

`target_resources` lists the identifiers of resources the mandate covers. `upstreams` tells the Gateway or SDK how to route the request and what credential to present to the upstream.

## What a mandate is not

A mandate is not a session cookie, an API key, or a long-lived secret. It expires within minutes, is bound to a specific zone, and resource mandates can be used exactly once via the Gateway. An agent that loses its mandate requests a new one only if its execution session, agent session, delegation edge, and policy still allow it.
