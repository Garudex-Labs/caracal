## Definition

A **Principal** is any identity that presents itself to Caracal and requests a mandate to act. Caracal recognizes three principal types:

- **Application** (`type: application`) — a service or agent process that authenticates with a signed client assertion (ES256 JWT) or a shared secret. This is the most common type. The Caracal SDK authenticates as an application principal on every `run` call.
- **Human user** (`type: user`) — a person acting through a Caracal-aware interface. Human principals authenticate via the coordinator's identity binding flow, typically through an OIDC provider.
- **Delegated agent** (`type: delegate`) — a sub-agent that received scoped authority from another principal via [Delegation](/concepts/delegation/). The delegate principal operates within the bounds the delegating principal granted.

Every principal belongs to exactly one [Workspace](/concepts/workspace/). The workspace's [Policy](/concepts/policy/) determines what that principal may do.

## Why it exists

Caracal needs a stable, verifiable identity for every authority request. Without a principal, there is no accountability: no one to bind the mandate to, no one to record in the [Ledger](/concepts/ledger/), no authority graph to reason about.

The principal concept also enables delegation. A human can delegate scoped authority to an AI agent. That agent can delegate a further-scoped subset to a sub-agent. The delegation graph is always traceable back to a named principal.

## Lifecycle

| Stage | Description |
|---|---|
| **Registration** | An application is registered in a workspace with its authentication method (assertion or shared secret). Human users are bound via the coordinator's `/sessions/bind` endpoint. |
| **Authentication** | On each `POST /oauth/2/token` call, STS verifies the principal's credential: it checks the assertion signature for application principals, or validates the bound session token for user/delegate principals. |
| **Active** | The principal is authenticated and may request mandates within its workspace's policy. |
| **Revoked** | A session revocation event on `caracal.sessions.revoke` invalidates all in-flight mandates for this principal. The gateway's revocation cache enforces this in real time. |

## Examples

**Application principal authentication — TypeScript SDK**

```typescript title="Authenticate as an application principal"
import { CaracalClient } from '@caracalai/sdk'

// The SDK authenticates as the application principal automatically.
// CARACAL_APP_ID and CARACAL_APP_SECRET (or assertion key path) identify the principal.
const client = new CaracalClient({
  coordinatorUrl: process.env.CARACAL_COORDINATOR_URL,
  gatewayUrl: process.env.CARACAL_GATEWAY_URL,
})

// Each run() call presents the application principal to the STS.
const session = await client.run({
  intent: { action: 'file:read', resource: '/data/reports/*.csv' },
})
```

**Delegate principal — receiving scoped authority**

```typescript title="Receive a delegation and act as a delegate principal"
import { CaracalClient } from '@caracalai/sdk'

const client = new CaracalClient({
  coordinatorUrl: process.env.CARACAL_COORDINATOR_URL,
  gatewayUrl: process.env.CARACAL_GATEWAY_URL,
})

// The delegation token was issued to this agent by an upstream principal.
// The agent now acts as a delegate principal scoped to what was delegated.
const session = await client.fromDelegation({
  delegationToken: process.env.DELEGATION_TOKEN,
})
```

## Constraints and limits

- A principal cannot request a mandate for an action that its workspace policy does not permit, regardless of how the authentication succeeds.
- Delegation cannot expand authority: a delegate principal can never hold more authority than the principal that delegated to it.
- A principal's workspace assignment is immutable after registration. To change workspaces, re-register the application.
- Shared-secret authentication is supported for development and low-trust environments. Production deployments use signed assertions.
