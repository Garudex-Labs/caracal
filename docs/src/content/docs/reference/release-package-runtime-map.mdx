Caracal uses one product release train and language-specific package versions. Use this page to confirm which installed artifacts belong together.

## Current release train

| Surface | Version or tag | What it covers |
| --- | --- | --- |
| Product release | `2026.05.27-rc.1` | Release manifest, binary archives, container images, Helm chart, and validation artifacts. |
| Runtime and Console binaries | `v2026.05.27-rc.1` release assets | `caracal` and `caracal-console` archives for supported operating systems. |
| npm packages | `0.1.3-rc.1` | TypeScript SDK, connectors, admin client, OAuth, identity, revocation, and core packages. |
| PyPI packages | `0.1.3rc1` | Python SDK, connectors, OAuth, identity, revocation, and core packages. |
| Go modules | Source tag `v2026.05.27-rc.1` | Go SDK, connectors, OAuth, identity, revocation, and core packages are consumed from the repository tag. |
| Helm chart | Product release `2026.05.27-rc.1` | Chart and values under `infra/helm/caracal`. |

## Runtime binaries

| Binary | Purpose | Platforms |
| --- | --- | --- |
| `caracal` | Local stack lifecycle, readiness, purge, token-injected workload execution, and Console launch. | Linux x64, Linux arm64, macOS x64, macOS arm64, Windows x64 |
| `caracal-console` | Interactive setup, management, diagnostics, audit search, and audit explanation. | Linux x64, Linux arm64, macOS x64, macOS arm64, Windows x64 |

The install scripts download the matching binary archives for the selected release and verify them against the release checksums.

## Runtime containers

| Image role | Components |
| --- | --- |
| Product runtime | API, Coordinator, Control, runtime helper |
| Enforcement path | STS, Gateway, Audit |
| Data plane dependencies | PostgreSQL, Redis |

Use the product release tag for all runtime containers in a deployment. Do not mix containers from different product release tags unless an upgrade guide explicitly says the combination is supported.

## Compatibility rule

Keep the runtime stack, Console, SDKs, and connector packages on the same release train during onboarding and incident response. Upgrade the runtime stack first, then update SDKs and connectors after the target release is healthy and audit evidence confirms normal token exchange and Gateway behavior.
