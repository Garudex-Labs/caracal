Caracal accepts configuration from three input surfaces. When more than one is present, the loader applies a fixed precedence so the resolved configuration is deterministic. This page documents that order for both the runtime shell (`caracal up`, `caracal run`) and the SDK factories (`Caracal.connect()`, `from_env`, `from_config`).

## Runtime shell precedence

`caracal run` and `caracal up` resolve runtime configuration through `loadRuntimeConfig()` in `packages/engine/src/runtimeConfig.ts`. The order is:

1. **`$CARACAL_CONFIG` file** — if set and the file exists, it is parsed as TOML and used as the sole source. Environment variables are not read after this. If the variable is set but the file is missing, the load fails when configuration is required.
2. **Environment variables** — if `$CARACAL_CONFIG` is unset and `CARACAL_*` runtime variables are present, configuration is built from them.
3. **Default file location** — if neither of the above produced a configuration, the loader falls back to the platform default (`$XDG_CONFIG_HOME/caracal/caracal.toml`, otherwise `~/.config/caracal/caracal.toml`).

The first surface that resolves to a valid configuration wins. The remaining surfaces are not merged on top of it.

## SDK precedence

The SDK exposes a single entry point that mirrors the runtime resolution:

- **TypeScript / Go** — `Caracal.connect()` / `caracal.Connect()` dispatch to `fromEnv()` / `FromEnv()`, which select between subject-token mode (`CARACAL_SUBJECT_TOKEN`) and client-secret mode (`CARACAL_APP_CLIENT_SECRET` + `CARACAL_STS_URL`) based on which variables are set. Pass explicit `ClientSecretOptions` to `connect()` to bypass env discovery.
- **Python** — `Caracal.connect()` additionally honours `caracal.toml`:
  1. Explicit `config_path=` argument → `from_config(path)`.
  2. Default `caracal.toml` at `$XDG_CONFIG_HOME/caracal/caracal.toml` (or `~/.config/caracal/caracal.toml`) → `from_config()`.
  3. Otherwise → `from_env()` with the same env-variable resolution as TS/Go.

`from_env()` and `from_config()` remain as explicit escape hatches when an application needs to pin a single resolution path.

## Variables consumed in each mode

| Mode | Required | Optional |
| --- | --- | --- |
| Subject-token | `CARACAL_COORDINATOR_URL`, `CARACAL_ZONE_ID`, `CARACAL_APPLICATION_ID`, `CARACAL_SUBJECT_TOKEN` | `CARACAL_GATEWAY_URL`, `CARACAL_RESOURCES` |
| Client-secret | `CARACAL_COORDINATOR_URL`, `CARACAL_ZONE_ID`, `CARACAL_APPLICATION_ID`, `CARACAL_APP_CLIENT_SECRET`, `CARACAL_STS_URL` | `CARACAL_APP_RESOURCES`, `CARACAL_RESOURCES`, `CARACAL_GATEWAY_URL` |
| `caracal.toml` (Python only) | `zone_id`, `application_id`, `app_client_secret` (or `app_client_secret_file`), `sts_url`, `coordinator_url`, at least one resource binding | `gateway_url` |

## Diagnosing the resolved source

The runtime shell records the source it loaded from (`environment`, the TOML path, or `default`) in startup logs. When debugging configuration disputes, capture that line first — it identifies which surface was applied and what the remaining surfaces would have contributed.
